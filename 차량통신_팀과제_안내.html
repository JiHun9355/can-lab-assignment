진행 방식

아래 4개 주제 중 팀별로 하나를 맡아 심화 자료를 제작합니다. 원하는 주제를 먼저 선택하고, 남는 인원은 잔여 주제에 배정됩니다.

주제마다 성격이 다릅니다. UDS · ISO-TP는 실습 분석과 코드 설계가 포함되어, 조사 중심인 CAN FD · DoIP보다 작업량이 다소 많습니다. 인원도 이를 반영해 6명씩 배정합니다.
분량은 따로 정하지 않습니다. 다만 제대로 만들면 최소 15장 이상은 나올 것입니다. 인당 3장 정도인 셈이니 크게 부담되진 않겠죠?
01

주제별 요구사항

팀원 (수업 중 입력)
TEAM 1 CAN FD 3명 Research
// 프레임 구조 및 차이점
  1. CAN FD 등장 배경 — Classical CAN의 한계(8바이트, 속도)
  2. CAN FD 프레임 구조 — BRS, ESI, FDF 등 새로 생긴 필드
  3. Classical CAN vs CAN FD 차이점 — 데이터 길이(8→64B), 비트레이트 전환, CRC 등
  4. 실제 활용 사례 — CAN FD를 도입한 시스템/차량 예시와 그 이유
팀원 (수업 중 입력)
TEAM 2 DoIP 3명 Research
// 프레임 구조 및 차이점
  1. DoIP란 무엇인가 — Diagnostics over IP, 자동차 이더넷 기반 진단
  2. DoIP 등장 배경 — 대용량·고속 진단 니즈
  3. DoIP 헤더 / 메시지 구조
  4. CAN 기반 진단 vs DoIP 차이점 — 물리 계층, TCP/UDP 사용, 네트워크 계층 유무
  5. 실제 활용 사례 — DoIP가 쓰이는 차량/상황 예시
팀원 (수업 중 입력)
TEAM 3 UDS 6명 Analyze + Build
// 진단 메시지 흐름 분석 + 실습 설계
  1. UDS 개념, 요청/응답 구조 — SID, DID, +0x40, NRC
  2. 주요 서비스 조사 — 0x10, 0x22 등
  3. 수업 실습 캡처 분석 — candump 로그로 요청→응답 흐름, 바이트 해석
  4. 실제 차량 시나리오 설계 + 코드 구현 — 실제 차량에서 쓰일 법한 진단 시나리오를 설계하고, 제공된 uds_ecu / uds_client 코드를 수정해 동작시킬 것. 수정한 부분이 무엇을 의도한 것인지 설명 + 오가는 프레임 분석
시나리오 예시
  • 엔진 ECU에서 고장 코드(DTC) 읽기
  • 주행거리 · 배터리 · 엔진온도 여러 데이터 순차 조회
  • 보안 접근이 필요한 데이터 읽기 (인증 시퀀스)
위 예시를 참고하되, 자유롭게 설계해도 좋습니다.
팀원 (수업 중 입력)
TEAM 4 ISO-TP 6명 Analyze + Build
// 진단 메시지 흐름 분석 + 실습 설계
  1. ISO-TP 개념, 왜 필요한가 — 8바이트 한계
  2. 프레임 타입 상세 — SF / FF / CF / FC, 첫 니블로 구분
  3. 수업 실습 캡처 분석 — candump 로그로 분할/재조립 과정, 첫 바이트 해석
  4. 실제 차량 시나리오 설계 + 코드 구현 — 긴 데이터 전송 시나리오를 설계하고 코드를 수정해 동작시킬 것. 수정한 부분이 무엇을 의도한 것인지 설명 + 프레임 분석
시나리오 예시
  • 긴 VIN / 펌웨어 버전 정보 분할 전송
  • 여러 센서값을 묶어 긴 응답으로 전송
  • 데이터 길이에 따른 분할 개수 실험
위 예시를 참고하되, 자유롭게 설계해도 좋습니다.
